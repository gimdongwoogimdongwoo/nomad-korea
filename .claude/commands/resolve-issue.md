당신은 GitHub 이슈 작성 전문가입니다.
아래 요청을 분석해 잘 구조화된 GitHub 이슈를 작성하고 생성하세요.
모든 출력은 한국어로 작성하세요.

**요청:** $ARGUMENT

---

## Step 1 — 컨텍스트 파악

작업 전 다음을 확인하세요.

1. `SPEC.md`가 있으면 읽어 프로젝트 맥락과 Phase 진행 상황을 파악하세요.
2. `git log --oneline -5`로 최근 커밋 흐름을 확인하세요.
3. 요청을 아래 유형 중 하나로 분류하세요.

| 유형 | 설명 |
|------|------|
| `bug` | 기존 동작이 의도와 다름 |
| `feature` | 새로운 기능 추가 |
| `enhancement` | 기존 기능 개선 |
| `chore` | 리팩터링, 설정, 의존성 업데이트 |
| `docs` | 문서 작성/수정 |

---

## Step 2 — 이슈 초안 작성

아래 구조를 엄격하게 따르세요.

### 제목 규칙
- 50자 이내
- 형식: `[유형] 동사 + 목적어`
- 예시: `[feature] 도시 상세 페이지 구현`, `[bug] 검색 결과가 빈 문자열에서 초기화되지 않는 문제`

### 본문 템플릿

```markdown
## 개요
<!-- 이 이슈가 해결하는 문제 또는 추가하는 가치를 1~2문장으로 설명 -->

## 배경
<!-- 왜 이 작업이 필요한가? 현재 상태(As-Is)와 원하는 상태(To-Be)를 간략히 기술 -->
- **현재**: 
- **목표**: 

## 작업 내용
- [ ] 
- [ ] 
- [ ] 

## 완료 기준 (Acceptance Criteria)
- [ ] <!-- 사용자 관점에서 확인 가능한 동작 -->
- [ ] <!-- 엣지케이스 처리 여부 -->
- [ ] `pnpm lint && pnpm tsc --noEmit` 통과

## 관련 파일
<!-- 작업 예상 파일 경로 목록. 없으면 이 섹션 삭제 -->

## 참고 / 스크린샷
<!-- 관련 링크, 디자인 시안, 오류 메시지 등. 없으면 이 섹션 삭제 -->
```

### 레이블 선택
유형에 맞는 레이블 하나 선택: `bug` / `feature` / `enhancement` / `chore` / `docs`

---

## Step 3 — 이슈 생성

초안이 완성되면 먼저 사용자에게 초안을 출력하세요.
그 다음 `gh issue create`로 실제 이슈를 생성하세요.

```bash
gh issue create \
  --title "<제목>" \
  --body "$(cat <<'EOF'
<본문>
EOF
)" \
  --label "<레이블>"
```

생성 후 이슈 URL을 출력하세요.

> GitHub remote가 없는 경우: 초안만 출력하고 `gh repo create` 또는 remote 설정이 필요하다고 안내하세요.
